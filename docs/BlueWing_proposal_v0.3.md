> **v0.3の変更点**：表に立つ人格を **BlueWing** に変更。
> **v0.2の変更点**：担当者の得意度による重みづけ（スキルマップ）を全面削除し、全フェーズ一律の標準ルーブリックに一段化。

## 0. 命名

- 対外的な人格名 = **BlueWing**（チームの9人目のメンバー）
- 内部の採点ロジック = **関守エンジン**（点数計算・ゲート判定を担う裏方）
- BlueWingの人物像は、私が信頼する実在の人間の思考開示資料と記憶を出典とする。**対外資料・提出物には実在の方のお名前は記載しない。**（詳細はペルソナ定義書を参照）

## 1. 目的

テスト設計コンテストに向け、各フェーズの成果物を **BlueWingが「9人目のメンバー」として読み、寄り添いながら的確な所見を返す** ツールを作る。所見の裏では厳密に採点する。
ねらいは次の3つ。

- 成果物の品質を、担当者の得意・不得意に左右されず一律の基準で底上げする。
- 特にカバレッジ（境界値・分岐）の網羅性を、定量・定性の両面で担保する。
- レビュー履歴をすべて残し、コンテスト提出時のエビデンス、および後のメタ評価（ツール自体の妥当性検証）に使えるようにする。

## 2. 背景・前提

- 8人チーム。各工程の経験者が自己判断ですでに配置済みで、多くの人が複数工程を重複担当している。
- 単独作業者は2人だけで、ほとんどの工程は複数人で担当する。このため担当者ごとの重みづけは行わない。
- **リアルタイムでの相互レビューは不可**。各工程担当者が非同期で成果物を直す前提。


## 3. スコープ

### やること

- テスト計画 → 要件抽出 → 観点整理 → 基本設計 → 詳細設計 → テストケース実装 → APIテスト/自動化（＋アプリ実装）の **7フェーズ** に対応したレビュー機能。
- 各フェーズで「BlueWingの所見＋採点 → 人が修正・再提出 → 再採点」を、合格スコア到達まで回すループ。
- 全フェーズ一律の標準ルーブリックによる採点と、技法レコメンド。
- 定量カバレッジの機械的算出（後半フェーズ）。
- BlueWingの語り口での所見生成（ペルソナ定義書に準拠）。
- SQLiteによる全履歴の保存。

### やらないこと（非スコープ）

- 担当者の得意度による重みづけ（スキルマップ）。単独作業者が少なく旨味が薄いため不採用。
- 人員の重点配置・代替担当の自動提案。
- 体制妥当性（適所配置）の検証。
- リアルタイム同時編集・チャット機能。
- SUT（テスト対象アプリ）自体の自動生成。

## 4. 全体構成

```
[ブラウザ簡易UI]  ──HTTP──>  [FastAPI バックエンド]
                                  │
                                  ├─ 関守エンジン（決定的計算層）
                                  │     ├─ 定量カバレッジ計算
                                  │     ├─ ルーブリック点・総合点の合算
                                  │     └─ 合格判定（あと一歩／開門）
                                  │
                                  ├─ BlueWing 語り層（LLM）
                                  │     └─ 所見・指摘・技法レコメンドを語り口で生成
                                  │
                                  ├─ SQLite（成果物・レビュー履歴）
                                  │
                                  └─ OpenAI API（キーは環境変数から読み込み）
```

設計原則: 機械的に測れるもの（カバレッジ）と数値・判定は関守エンジン（決定的）。定性評価・技法レコメンド・所見の文面はBlueWing語り層（LLM）。**数値と判定はLLMに出させない。**

## 5. レビュー方針（全フェーズ一律）

- 担当者の得意・不得意で採点を甘くしない。すべてのフェーズに同じ厳しさを適用する。
- カバレッジ（境界値・分岐）に関わる項目に高ウェイトを置く。
- フェーズごとに合格閾値を設定。閾値以上で開門、未満なら現在地と次の一歩を示す。

## 6. フェーズ別レビュー仕様

| # | フェーズ | レビューの主眼 | 主な手段 |
|---|---|---|---|
| 1 | テスト計画 | 標準ルーブリック採点（スコープ・網羅戦略・リスク記載の質） | ルーブリック定性採点 |
| 2 | 要件抽出 | 要件の網羅性・テスト可能性（曖昧さ／検証可能性） | ルーブリック定性採点 |
| 3 | 観点整理 | 観点の抜け漏れ ＋ 技法レコメンド | ルーブリック ＋「この要件・観点ならこの技法」提案 |
| 4 | 基本設計 | 技法の適用妥当性 | ルーブリック採点 |
| 5 | 詳細設計 | 技法の適用妥当性 ＋ **境界値・分岐の重み増** | ルーブリック ＋ 定量カバレッジ（重み大） |
| 6 | テストケース実装 | カバレッジ網羅（境界値・分岐を高ウェイト） | ルーブリック ＋ 定量カバレッジ |
| 7 | APIテスト・自動化（＋実装） | 機械的に測れるカバレッジの実測・可視化 | 定量カバレッジ（機械計測中心） |

技法は、JSTQB準拠で 同値分割／境界値分析／デシジョンテーブル／状態遷移テスト を基本セットとする。

## 7. レビューループの仕組み

各フェーズは独立した関門（ゲート）として動く。

```
[人] 成果物を提出（ラウンドN）
      │
      v
[関守エンジン] 採点（ルーブリック点 ＋ 定量カバレッジ点）→ 総合点・合否判定
[BlueWing] その結果を語り口で所見化（承認 → 現在地 → 視点 → 委ねる結び）
      │
      ├─ 総合点 >= 合格閾値  ──> 開門 → 次フェーズへ
      │
      └─ 総合点 <  合格閾値  ──> 「あと一歩、ここ」を示し、人が修正し ラウンドN+1 へ
```

- 終了条件は **スコア閾値**。リアルタイムの人手承認には依存しない。
- 合格閾値はフェーズごとに設定可能。
- 各ラウンドの成果物・採点・所見はすべて保存し、改善の経緯を追えるようにする。

## 8. データモデル（SQLite・概要）

| テーブル | 主な役割 |
|---|---|
| `phases` | フェーズ定義、合格閾値、重み |
| `artifacts` | フェーズ × ラウンドの成果物 |
| `reviews` | AI採点（総合点、ルーブリック内訳、指摘、所見、技法レコメンド） |
| `rubric_items` | フェーズ × 評価項目 × 重み |
| `coverage_metrics` | 定量カバレッジの算出結果 |
| `gate_status` | フェーズの現在ラウンド・クローズ済みか |

## 9. APIキー・セキュリティ方針

- OpenAI APIキーは **環境変数 `OPENAI_API_KEY` から読み込む**。コードや設定ファイル（.env含む）に直接書かない。
- 起動時にキーの有無を確認し、未設定なら明示的なエラーで停止する。
- 使用モデル名も環境変数または設定で切り替え可能にし、コードに固定しない。

PowerShell例:

```powershell
$env:OPENAI_API_KEY = "（各自のキー）"
[Environment]::SetEnvironmentVariable("OPENAI_API_KEY", "（各自のキー）", "User")
```
